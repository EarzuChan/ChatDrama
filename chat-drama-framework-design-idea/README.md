# Earzu's Chat Drama

**Chat Drama** 是一个面向“模拟手机 IM 聊天界面”的 AI RPG / Galgame / 推理解谜叙事框架。

它的核心不是“一个角色一个聊天机器人”，而是：

> **一个单主控用户，通过一台模拟手机，与多个聊天会话中的多个 NPC 角色互动；系统以 Phase（一拨对话）为叙事单位，以 Beat 为期内拍点，以 Transaction 为状态提交单位，由 Director / Actor / Validator / Memory 等 Agent 协作，驱动剧情、角色记忆、线索、分支和状态。**

---

## 当前设计边界

### 做什么

- 单主控用户
- 纯文本 IM 聊天
- 单聊 / 群聊
- 多 NPC 角色
- 角色人设、状态、记忆、知识隔离
- 世界状态、线索、剧情分支、路线
- Phase 制对话编排
- 用户可在显现期间打断
- 前台会话驱动剧情
- 后台聊天可被 bump 一期
- 存档 / 读档
- 调试暂停 / 覆写 / 回放
- LLM API 与游戏引擎解耦

### 目前明确不做 / 未计划

- 多用户 / 联机。为什么不做：麻烦，且很多RPG都是单主角（真人用户）
- 图片、语音、视频、文件，也即更多的消息类型：很多模型都有多模态，读取更多类型的文件本身不是问题，问题在于：文件被限制为戏内的资产（剧情上主角可获取的Item），还是允许任意提供？
- 消息撤回：没设计这样的机制
- 角色离线 / 在线状态：还没设计这样的机制，但未来可以设计
- 多个会话同时作为主会话运行：那会导致现有模型要改造，且在这种情况下，回复时间窗口不利好用户，也需要重新设计
- 戏内时间回溯：不是所有剧情都适合戏内时间回溯，故当方向严重偏离约束方向时，应使得游戏终结，让用户自行回档
- 真实物理时间自动推进剧情：同理，不利好用户和游戏开发。RPG的时间推进，一般以离散时间（事件驱动）为主

---

## 核心一句话

Chat Drama 不是以“单条消息”为单位运行，而是以 Phase 为单位编排，以 Beat 为单位组织角色发言，以 Transaction 为单位提交状态，以 Response Window 为单位把控制权交还给玩家。

## 适用类型

### 本格推理

可支持：

- 线索收集
- 玩家推理验证
- 角色证词
- 真相锁
- 证据链
- 死档 / 读档

### Galgame / 恋爱 ADV

可支持：

- 多角色线
- 私聊与群聊
- 好感、信任、紧张度
- 关键事件门槛
- 路线锁
- 结局条件

### 群像 RPG

可支持：

- 多角色对话
- 世界状态变化
- 后台聊天 bump
- 多线 storylet
- 角色记忆与误解
