# LLM 包的设计理念

本包提供一套统一 LLM API，并用后端适配四类提供商形态：OpenAI Chat Completions 兼容后端、OpenAI Responses、Claude、Gemini。上层代码面对的是统一模型；底层的各后端尽量贴近其提供商的本家协议。

严正提示：我们这个包非常的面向未来、非常的打破传统，故我们的范式需要你深度理解，切莫拍脑子混淆旧概念！

## Session 与 Turn

传统接口常把一次完整模型调用约束成 `prompt -> answer`。这里不采用这个老范式，而是使用我们设想的 `Turn` 理念：一次请求之后，只要模型返回了内容，就构成一次结果；结果由多种类型的项目组成 `items`，可以是思考、正文、工具调用请求、拒绝等。

于是，我们这种设计允许：

- 思考和正文混排：这一点天生更好地面向了如Claude的正文内联思考的形式。
- 一轮能以工具调用请求作为结束：因为这是一轮。一轮是设计为一次的网络请求，而老范式的“完整一次”往往可能涉及多次网络请求。
- 平行工具调用请求自然存在于结果的 items 中，模型方想调用同时多少个工具都不是问题。
- 用户自行决定如何处理工具调用、如何继续下一步，而不是被包的设计模式绑架。

因此，这个包不要求用户先注册一套高级工具系统，再由包内部自动跑完“工具调用循环”。相反，它暴露足够细的结构，让用户可以自行包装任意 agentic 模式。LLM 不再被限制为只能是一个 `assistant`，而可以轻松成为系统流程中的一环。

## 会话的双轨制：Lane A 与 Lane B

`Session` 使用双轨制。

`Lane A` 是本地完整上下文，保存全部节点、全部返回 items、全部黑板信息（黑板的概念借鉴自游戏开发领域）。黑板按 session、root、node 层叠承载任意细节，便于回看、调试、分叉、编辑和审计。

`Lane B` 是提供商原生上下文缓存。它面向实际发送请求，保存各后端自己的原生消息、输入、响应状态或压缩结果。请求成功后，提供商返回的原生内容直接追加到 Lane B，同时再解析成 Lane A 的统一语义提交给 Session。

这个设计有这些效果：

- 本地能完整保留全部细节（于 Lane A），不因模型压缩或提供商限制丢历史，且细节是细节到：留存着提供商的私有细节。
- 供给提供商的，不必每次从零重建完整上下文（只要缓存状态有效，那提供商上下文（Lane B）是采用追加的模式），性能和实用性更好。
- Lane B 可以承接各厂独特的上下文状态与压缩机制。
- 压缩只影响模型侧上下文（能省钱/使得模型状态更好），不触及本地完整上下文（Lane A）。

当 Lane B 失效或切换提供商时，会从 Lane A 重建。重建前会先打薄已闭合的完整波（波：指老范式中一次完整的对话，即 `prompt -> answer`）：保留开头用户请求和结尾可见回复，去掉中间思考、工具调用等过程细节；未闭合的波（没有完整的 `prompt -> answer` 闭环）不会被打薄。各后端追加原生新内容后，也会检测最新波是否闭合，闭合则在 Lane B 内直做本家小压缩。

## 配置与跨平台

调用配置支持即时覆盖，包括模型、缓存偏好、思考程度、输出约束、温度和各提供商独有选项。统一配置只表达公共意图，用户想搞特殊，仍可通过 provider options 设入提供商后端。

后端默认尽量申请厂商跨域相关能力，这很利好我们的 Kotlin Multiplatform：在浏览器、Wasm、JVM 等环境都追求尽量能用，减少用户的多平台操心。
